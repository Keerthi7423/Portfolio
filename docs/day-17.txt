OVERVIEW:
Today's mission focused on implementing Phase 6: "Battle History". We built a 
cinematic, scroll-driven vertical timeline that tracks the developer's career 
milestones, styled as mission logs from high-clearance agencies like S.H.I.E.L.D. 
and Stark Industries.

--------------------------------------------------------------------------------
1. KEY FEATURES IMPLEMENTED
--------------------------------------------------------------------------------

A. DYNAMIC VERTICAL TIMELINE (Framer Motion + Scroll)
   - A glowing central spine that "draws" itself as the user scrolls.
   - Used Framer Motion's `useScroll` and `useTransform` to map scroll progress 
     to the height of the timeline line.
   - Includes a gradient pulse effect (Arc Blue to Marvel Red) representing 
     energy flow through the career history.

B. STAGGERED REVEAL CARDS
   - Experience cards slide in from the left (even) or right (odd) using 
     viewport-aware animations.
   - Implemented a custom easing function `[0.16, 1, 0.3, 1]` (EaseOutExpo) 
     for a premium, cinematic feel.
   - Cards feature scanline overlays and holographic glow effects on hover.

C. THEMATIC DATA ARCHITECTURE
   - Created `lib/experience-data.js` to separate content from logic.
   - Integrated custom metadata like "LOG_ID" and "MISSION_OBJECTIVES" to 
     maintain the Stark Industries UI narrative.
   - Added a pulsing "ACTIVE" indicator for the current role.

D. RESPONSIVE HUD LAYOUT
   - Rotating background HUD elements (CSS/GSAP) for depth.
   - Adaptive timeline: On mobile, the timeline shifts to the left, and cards 
     stack vertically for optimal readability.

--------------------------------------------------------------------------------
2. TECHNICAL CHALLENGES & SOLUTIONS
--------------------------------------------------------------------------------

CHALLENGE: Synchronizing Timeline Growth with Scroll
Standard CSS transitions for height can feel laggy and don't correlate with 
user effort.
SOLUTION: We used Framer Motion's `useSpring` on top of `scrollYProgress`. This 
adds "inertia" and "stiffness" to the growth, making the line feel physical 
rather than digital.

CHALLENGE: Maintaining Performance with Multiple Cards
Animating 10+ complex cards with blur and gradients can drop frame rates.
SOLUTION: Used `whileInView` with a `margin` offset so animations only trigger 
when elements are near the viewport. We also leveraged `backdrop-blur-md` 
carefully, ensuring it only applies to the surface layer.

--------------------------------------------------------------------------------
3. INTERVIEW PREPARATION (Q&A)
--------------------------------------------------------------------------------

Q: How did you implement the "drawing" effect of the timeline line?
A: I used the `useScroll` hook from Framer Motion to track the scroll progress 
of the section. I then mapped that 0-1 value to a height percentage using 
`useTransform`. To make it smoother, I passed the scroll progress through a 
`useSpring` hook, which adds a natural physics-based lag (stiffness/damping), 
creating a more high-end feel.

Q: Why use GSAP for background rotations but Framer Motion for the timeline?
A: GSAP is superior for infinite, high-performance rotations and non-scroll 
related timelines (like the HUD elements). Framer Motion is much better 
integrated with React's state and viewport-based triggers (like `whileInView`), 
making it the better choice for the layout and entrance animations of the cards.

Q: How do you handle mobile responsiveness for a centered timeline?
A: I used a "Mobile-First" approach with Tailwind. On mobile, I absolute-position 
the timeline to the left (`left-[20px]`) and remove the `isLeft` logic so all 
cards align to the right of the line. On `md` screens and above, I shift the 
line to the center (`left-1/2`) and alternate the card placement.

Q: How do you optimize images for a timeline with many logos?
A: I used Next.js's `next/image` component. For logos, I set specific `width` 
and `height` (e.g., 64px) to ensure the browser doesn't load massive source 
files. I also used a grayscale-to-color transition on hover using CSS filters, 
which is handled by the GPU and doesn't require swapping image files.
