1. KEY FEATURES IMPLEMENTED
--------------------------------------------------------------------------------

A. THE "SCANNING BEAM" OVERLAY
   - Implemented a `ScanningEffect` component that triggers when an Experience 
     Card enters the viewport.
   - Uses a horizontal light beam (linear-gradient with glow) that moves 
     vertically down the card, simulating a "biometric scan" of the dossier.
   - Powered by `framer-motion`'s `whileInView` prop with a single-run transition.

B. STAGGERED MISSION OBJECTIVES
   - Refined the `ObjectiveList` component to use a more precise stagger 
     algorithm.
   - Each bullet point (achievement) now slides in from the left with a unique 
     delay: `0.6 + (index * 0.15)`.
   - Added `Target` icons and glowing bullet points to reinforce the "Mission 
     Objective" aesthetic.

C. "LIVE MISSION" STATUS INDICATOR
   - Enhanced the "Active" role marker into a Stark-style HUD badge.
   - Includes a dual-layer ping animation (core + ripple) and a text label 
     ("LIVE MISSION") in the `Orbitron` font.
   - This provides immediate visual hierarchy for the user's current occupation.

D. CINEMATIC TEXT ANIMATIONS
   - Implemented `initial={{ opacity: 0 }}` and `whileInView={{ opacity: 1 }}` 
     for role titles and descriptions.
   - Added "Decorative Corner Brackets" and "Scanning Lines" (via CSS gradients) 
     to every card to maintain the technical/classified look.

E. HUD HEADER ENHANCEMENT
   - Updated the `SectionHeader` with a "Transmission Active" blinking status

   - This ties the entire section together as an active data stream from 
     Stark Industries servers.

--------------------------------------------------------------------------------
2. TECHNICAL CHALLENGES & SOLUTIONS
--------------------------------------------------------------------------------

CHALLENGE: Performance of Concurrent Animations
Running multiple "Scanning Beam" animations and staggered list reveals 
simultaneously can be heavy on the main thread.
SOLUTION: Leveraged `framer-motion`'s `viewport={{ once: true }}` to ensure 
heavy entrance animations only run once per session. This prevents redundant 
calculations as the user scrolls back and forth, keeping the FPS stable.

CHALLENGE: Responsive HUD Layouts
Maintaining the "Battle Log" look on mobile without cluttering the screen.
SOLUTION: Implemented conditional rendering for secondary HUD elements (like 
the "Security Clearance" tag) and used `flex-col` for mobile meta-details 
while keeping `md:items-end` for desktop to maximize whitespace.

CHALLENGE: Timing the Staggered Reveal
If the list reveal starts too early (before the card is visible), the user 
misses the animation.
SOLUTION: Used a combination of `whileInView` on the parent container and a 
coordinated `delay` on the children. The scan starts at 0s, the title at 0.3s, 
the description at 0.5s, and the objectives start staggered from 0.6s.

--------------------------------------------------------------------------------
3. INTERVIEW PREPARATION (Q&A)
--------------------------------------------------------------------------------

Q: What is a "Staggered Reveal" and why is it used in UI?
A: A staggered reveal is a technique where multiple elements (like list items) 
animate in one after another rather than all at once. It reduces "cognitive load" 
by guiding the user's eye through the content sequentially and adds a premium, 
polished feel to the interface.

Q: How did you implement the "Scanning Beam" effect without using images?
A: I used a simple `motion.div` with a height of 1px and a background gradient. 
By applying a `blur-sm` and a `box-shadow` with low opacity, I created a "glow" 
effect. The motion is handled by animating the `top` property from `-10%` to 
`110%` using a linear ease.

Q: Why use `framer-motion` over standard CSS animations for this section?
A: `framer-motion` provides much better "Viewport Awareness" out of the box. 
Triggering animations based on when an element enters the screen is much 
simpler and more robust than manually calculating scroll offsets with raw 
JavaScript. It also handles "Spring" physics much better than CSS transitions.

Q: How do you handle "Active" states in a dynamic data-driven component?
A: In the `experience-data.js`, we have a `status` field. In the component, I 
conditionally render the pulsing HUD badge using a simple logical AND: 
`{exp.status === "ACTIVE" && <StatusBadge />}`. This ensures the UI stays in 
sync with the underlying data source.
