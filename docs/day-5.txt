================================================================================
  DAY 5: FIXED NAVIGATION SYSTEM & HUD INTERFACE
================================================================================

OVERVIEW:
Today's mission was to build the primary control interface—the Navigation System.
Designed to mimic a Stark Industries HUD (Heads-Up Display), the Navbar remains 
fixed at the top, transitioning from a stealth (transparent) mode to an active 
(glassmorphism) state upon scrolling.

--------------------------------------------------------------------------------
1. TECHNICAL ARCHITECTURE: THE NAV SYSTEM
--------------------------------------------------------------------------------

- FIXED POSITIONING: The Navbar uses `fixed top-0` to remain accessible during 
  long vertical scrolls, ensuring the "Control Center" is always reachable.
  
- SCROLL-DRIVEN LOGIC: 
  - Implementation: Used a custom `useEffect` hook with a window scroll event 
    listener.
  - State: `isScrolled` boolean triggers at a threshold of 50px.
  - Visual Feedback: When `isScrolled` is true, the `.stark-glass` class is 
    applied, adding a backdrop-blur (12px) and a semi-transparent background 
    to maintain legibility over moving content.

- Z-INDEX ORCHESTRATION: Set to `z-[100]` to ensure navigation remains above all 
  3D elements and page sections, but below the JARVIS AI panel (Phase 7).

--------------------------------------------------------------------------------
2. COMPONENT DEEP DIVE: Navbar.jsx & MobileMenu.jsx
--------------------------------------------------------------------------------

A. THE LOGO ASSEMBLY:
   - Logic: A custom SVG-style React component using Tailwind's `rotate-45`.
   - Hover Effect: On hover, the outer frame rotates 180 degrees via `transition-700`, 
     simulating a mechanical lock/unlock sequence typical of Iron Man armor parts.

B. GLASSMORPHISM (STARK-GLASS):
   - CSS Utility: Defined in `globals.css`, it combines `bg-space/80`, 
     `backdrop-blur-md`, and a subtle `border-border/50`.
   - Purpose: Creates a premium, high-tech aesthetic that feels integrated into 
     the 3D environment without obstructing it.

C. FULL-SCREEN MOBILE OVERLAY:
   - Transition: Powered by Framer Motion's `AnimatePresence`.
   - Reveal Logic: Uses a "Spring" transition (`damping: 30`) for a snappier, 
     more physical feel compared to linear ease.
   - Staggered Links: Each menu item slides in from the right with a 0.1s stagger 
     delay, creating a sequential reveal that guides the eye.

--------------------------------------------------------------------------------
3. INTERVIEW PREPARATION: "HOW WOULD YOU EXPLAIN THIS?"
--------------------------------------------------------------------------------

Q: Why use a "scroll threshold" for the Navbar transition instead of a toggle?
A: A threshold (like 50px) allows the Hero section to breathe in "stealth mode" 
   at the start. As soon as the user engages with the page, the UI solidifies 
   (glassmorphism), providing the necessary contrast for navigation links 
   against busier content sections.

Q: How do you handle mobile navigation in a "cinematic" way?
A: Instead of a simple dropdown, we use a full-screen overlay with staggered 
   animations. By animating the menu items one-by-one, we create a sense of 
   "system initialization." We also include decorative elements like scanlines 
   and emergency frequency numbers to maintain the narrative of a high-tech UI.

Q: How do you prevent scroll event listeners from killing performance?
A: In this implementation, the listener only updates a single boolean state. For 
   production, we could wrap the scroll handler in a `throttle` function (e.g., 
   from Lodash) to ensure the logic only runs every 10-20ms instead of on 
   every single pixel of movement.
